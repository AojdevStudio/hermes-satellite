Verification/Confidence tiers

Confidence tiers

Confidence is clamped in code by the strongest evidence tier available, not by how convincing the result sounds. The verifier reports a tier, the parser applies the cap, and file or code work cannot be marked verified without T2 evidence.

Clamp, do not trust vibes

The satellite report may propose PERFECT, VERIFIED, PARTIAL, FEEDBACK, or FAILED. The implementation then clamps that value by EVIDENCE_TIER. A strong paragraph in hermes_result cannot lift a weak evidence tier.

textCopy
reported = parse_report_confidence(report)
tier = parse_report_evidence_tier(report)
confidence = min(reported, cap_for(tier))

Tier caps

TierEvidence sourceCap
T0hermes_result final result text only.PARTIAL at best.
T1Bridge task row, session id, and hermes_sessions summary.VERIFIED at best.
T2Full transcript from hermes_transcript.PERFECT-eligible when all oracles pass.
T3Forced-output or live transcript stream with complete post-hoc record.PERFECT-eligible when complete.

Any task that writes files or executes code requires T2. If hermes_decompose returns a write, filesystem, or execution tool_execution atom, T0 and T1 cannot produce VERIFIED or PERFECT.

Structured claims

A schema-shaped assistant block is useful as a hint, but it is not canonical by itself. structured_assistant_claim caps at 0.55 confidence and cannot satisfy PERFECT unless the T2 transcript proves a forced output mechanism such as provider response format, strict tool use, a dedicated verify claims tool, or plugin-owned structured completion.

Cost confidence

Correctness confidence and cost visibility travel together. hermes_result should carry a cost block; hermes_task_cost is the fallback for the latest task cost or loop history. Report cost even when correctness is PARTIAL or FAILED.

i
MoA accounting

Mixture-of-agents rows can report local estimatedUsd as zero while real upstream spend happened. Surface those rows as unreconciled and never present them as a free authoritative $0.
