Verification/Decompose claims

Decompose claims

Verification starts by turning Hermes output into an AtomicClaim[]. Use hermes_decompose with the transcript or session id and the original prompt, then oracle the returned checklist instead of re-parsing the transcript by hand.

The decomposition contract

hermes_decompose(session_id or transcript, original_prompt) is the canonical entry point. The transcript supplies the T2 tool rows; the original prompt supplies the operator requirements that Hermes was asked to satisfy.

textCopy
result = hermes_result(task_id)
transcript = hermes_transcript(session_id)
claims = hermes_decompose(transcript, original_prompt)

The satellite verifier should receive a claim checklist, not raw transcript structure. Manual re-parsing makes every loop non-deterministic and lets the verifier miss requirements, double-count prose, or treat a summary as proof.

Why the original prompt matters

The dispatch prompt must include a ## Acceptance section with one testable bullet per requirement. hermes_decompose parses those bullets into deterministic user_requirement claims. Without the original prompt, the verifier can only infer intent from Hermes's answer, which is exactly the thing being checked.

i
Operator rule

Always pass the exact prompt that created the task. If the acceptance bullets are missing, say that requirement coverage is incomplete instead of inventing requirements from the transcript.

Claim taxonomy

KindWhere it comes fromOracle rule
tool_executionEach T2 tool_call plus matching tool_result.Compare only against embeddedEvidence from the tool result row.
assistant_assertionAssertion sentences from the final assistant message.Require a cited tool row or external oracle; never PASS on prose alone.
user_requirementBullets under the original prompt's ## Acceptance heading.PASS only when supporting tool evidence proves the requirement.
structured_assistant_claimSchema-shaped claim blocks in assistant output.Candidate only; cap confidence at 0.55 unless T2 proves a forced output mechanism.

Atomic claim shape

Each claim has an id, kind, source pointer, text, and optional evidence fields. tool_execution claims carry embeddedEvidence; structured_assistant_claim may carry structuredOutputMechanism when the transcript proves one.

The output exists to make verification boring: every row gets PASS, FAIL, or UNSURE, and no report can claim PERFECT while a returned claim is unchecked.
